A securities settlement system processes bond trades but produces incorrect net settlement amounts. Counterparties report discrepancies in their reconciliation, and the books don't balance. The symptoms manifest differently depending on the bond being traded - some bonds settle correctly while others show systematic errors in accrued interest calculations. Sell transactions also appear to have incorrect regulatory fee deductions compared to what counterparties expect based on current SEC rates.

The existing COBOL programs are located in /app/ and consist of four modules that work together: settle.cob serves as the main driver that reads trades and coordinates the calculation, daycount.cob handles date arithmetic for determining days between dates, accrued.cob computes accrued interest by looking up coupon schedules and bond characteristics, and secfee.cob calculates SEC Section 31 transaction fees for sell orders. These programs use copybooks from /app/copybooks/ including TRADEREC.cpy for trade input layout, BONDREC.cpy for bond reference data, COUPON.cpy for coupon schedule records, SECRATE.cpy for SEC fee rate data, SETTLOUT.cpy for settlement output records, and RULE606.cpy for SEC Rule 606 reporting. The input files reside in /app/data/ - trades.dat contains the trades to process, bonds.dat has bond reference data with day-count conventions and coupon frequencies, coupons.dat holds coupon payment schedules, and secrates.dat contains historical SEC fee rates.

Expected correct behavior requires that day-count conventions be honored per bond specification - bonds marked with convention "A" use actual/actual counting, "B" uses 30/360, and "C" uses actual/360. Accrued interest must be calculated from the most recent coupon payment date, respecting whether the bond pays quarterly (frequency 4) or semi-annually (frequency 2). SEC transaction fees must use the current effective rate from the rate file, not any hardcoded historical value.

The settlement output at /app/output/settlement.dat must contain fixed-length 45-byte records per the SETTLOUT.cpy layout with computed principal (quantity times price), accrued interest, SEC fee (zero for BUY trades), days accrued, and net settlement amount for each trade. For BUY trades, net settlement equals principal plus accrued interest. For SELL trades, net settlement equals principal plus accrued interest minus SEC fee. The SEC Rule 606 quarterly report at /app/output/rule606.dat aggregates order routing statistics per venue using the RULE606.cpy layout (30-byte records) - total orders, total shares, market vs limit order counts, and payment for order flow amounts.

When using COBOL programming language, all the source code must be located under /app/. All the copybooks need to be under /app/copybooks/. All the data files are to be under /app/data/. All the output files are to be under /app/output/. The code format must be fixed (coded between columns 7 through 72 only) and should not exceed 80 characters wide. You are not allowed to use the >>SOURCE FORMAT directives while coding. Compilation of your COBOL files must be done using GnuCOBOL v3. For example, if you want to compile a COBOL program named TEST, you would do it like this: cobc -x -I /app/copybooks -o /app/TEST /app/TEST.cob. If you have multiple programs as part of one task, you need to compile the modules using the -m flag (before compiling the main program) and include the COB_LIBRARY_PATH as /app for execution. You will be required to compile your code with no warnings and execute it successfully (exit code = 0). All output is to be generated in mainframe formats and must adhere to the correct fixed-length record format and COMP-3 numeric fields. You are not allowed to produce any JSON or CSV output.
