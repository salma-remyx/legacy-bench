The original Java 7 (JDK 1.7) Medical Claims Processing System for healthcare claims located in the /app/src/ folder utilizes JAX-WS with MTOM binding for managing claims which have been submitted through the SOAP protocol.

The Software consists of six (6) different classes, including "ClaimsProcessor.java" (which processes the batch), "ClaimsServiceImpl.java" (which serves as the @WebService and @WebMethod entry point), "ClaimValidator.java" (which validates member eligibility and provider status), "BenefitCalculator.java" (which determines the allowed amounts using spreadsheets of the plan’s benefits and fee schedules), "AttachmentProcessor.java" (which accepts binary Document Attachments) and "AuditLogger.java" (which tracks all events related to processing claims).

Attachments for claims are exhibiting issues. While the claim that contains an attachment less than 64KB will be properly processed and has a matching checksum, claims that have attachments greater than 64KB experience a loss of data (more so depending on the size of the document). The size after Processing is not the same as the original size and therefore the checksum calculated after Processing does not match what was initially submitted. The only acceptableSolution is to ensure that all Document Attachments remain intact (i.e., byte-for-byte), regardless of size during the Processing of the claim.

All reference Data (including CSV) files related to these claims exist in the /app/data/ folder including the following: claims.csv contains a list of all claim Lines filed, providers.csv has provider network status, members.csv has eligibility status, fee_schedule.csv includes information on allowable amounts, plan_benefits.csv includes information on deductibles and coinsurance as well as procedure_codes.csv and diagnosis_codes.csv for validation. Binary attachments are located in the /app/data/attachments/ folder as raw files.

The Input Files for the claims.csv file include the following columns: claim_id, member_id, provider_id, service_date, total_charges, line_id, procedure_code, diagnosis_code, charged_amount, units and the attachment_file name (in which to locate the document) or empty if no attachment was included.* The Providers.csv file contains the provider id, name, specialty, network status, effective date and termination date.
* The Members.csv file contains the member id, plan id, effective date, termination date, deductible met and out of pocket met.
* The Fee_schedule.csv file contains the procedural code, network allowed, out of network allowed and requires authorization for all fees charged for each member.
* The Plan_benefits.csv contains each plan's deductible, coinsurance rate, copay and out of pocket max.
* The procedural_codes.csv file will contain all codes associated with Service and each associated description, as well as all service related codes.
* The Diagnosis_codes.csv will contain each code associated with a diagnosis and the associated description.

* All amounts denoting money are expressed in integer format in cents, e.g. $200.00 = 20000
* All dates are in the format YYYY-MM-DD.
* A blank termination date indicates an active status.

The Claim_summary.csv output file will be located in /app/output/ and will contain the columns of claim id, claim status, error code, total charges, total allowed, total paid, patient responsibility, the number of attachments and attachment checksum, arranged in that exact order by claim id. The adjudication_details.csv file will contain the columns in the same exact order claim id, line id, procedure code, charged amount, allowed amount, paid amount, patient responsibility and adjustment reason sorted first by claim id and then by line id. Attachment_report.csv will contain the claim id, count of attachments and total checksum for claims with attachments. The Audit_log.csv file will contain logs of events concerning the processing. Compute attachment checksums using a polynomial rolling hash algorithm: starting with hash=0, for each byte b compute hash = (hash * 31 + b), then add the total byte count to get the final checksum. For example, for 150000 bytes of sequential data (bytes where byte[i] = i % 256 for i in range(150000)), the checksum is 1898254987636178664.

Use the Compiler (Java Compiler) to compile the source formation files to the /app/bin directory. Execute the claim processor program with the entry point that will accept 9 command line arguments: The name of the claims file, The attachments directory, The providers file, The members file, The fee schedule file, The plan benefits file, The procedural codes file, The diagnosis codes file and The output directory. When processing the attachments, all binary attachments data must be processed without errors or corruptions in an output file that is the same size and has the same hash as the original input file, for example, if you input a by the number of bytes equal to 100000 it must still be the same number of bytes in the output as well as having the same hash value.
